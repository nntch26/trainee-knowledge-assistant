# Architecture Decisions

## Decision 1: เลือกใช้ Next.js และ Express.js แยก Frontend และ Backend

### Context

โปรเจกต์นี้เป็นระบบ AI Chat ที่ต้องมีทั้งส่วนแสดงผลสำหรับผู้ใช้ ระบบยืนยันตัวตน การจัดการประวัติการสนทนา และการเชื่อมต่อกับ Google Gemini AI และการจัดการ Business Logic หลายส่วน จึงจำเป็นต้องมีสถาปัตยกรรมที่ชัดเจนและสามารถขยายระบบได้ง่ายในอนาคต

### Alternatives Considered

 การใช้ Next.js เป็น Full-Stack เพียงอย่างเดียวผ่าน API Routes ทำให้สะดวกและพัฒนาโปรเจกต์ได้ง่าย แต่การแยก Backend ออกมาเป็น Express.js โค้ดมีโครงสร้างที่ชัดเจนกว่า เวลาพัฒนาจะแยกส่วนๆ เป็นระบบมากกว่า

### Why Next.js + Express.js

Next.js เหมาะสำหรับการพัฒนา User Interface ทำให้โหลดหน้าเว็บได้ไว ส่วน Express.js มีความยืดหยุ่นสูงในการจัดการ REST API, Middleware และ Business Logic การแยกสองส่วนนี้ช่วยให้แต่ละระบบมีหน้าที่ชัดเจน และมีความคุ้นชินกับเครื่องมือมากกว่า เพราะเคยใช้ในการพัฒนาโปรเจกต์อื่นๆ

### Trade-offs

การแยก Frontend และ Backend ทำให้ต้องดูแลหลายโปรเจกต์และตั้งค่าการสื่อสารระหว่างระบบเพิ่มเติม เช่น CORS และ Environment Variables แต่ก็สามารถขยายระบบได้ในอนาคต ถ้ามีฟังก์ชันที่ซับซ้อนที่เพิ่มขึ้น

## Decision 2: การเลือกใช้ PostgreSQL ร่วมกับ Prisma ORM

### Context

ระบบต้องจัดเก็บข้อมูลผู้ใช้ ประวัติการสนทนา การใช้ Token ซึ่งเป็นข้อมูลที่ดูต้องมีความสัมพันธ์กัน จึงต้องใช้ฐานข้อมูลเชิงสัมพันธ์ที่มีความเสถียรและรองรับการขยายระบบได้ดี

### Alternatives Considered

ทางเลือกที่พิจารณา การใช้ MongoDB มีความยืดหยุ่นสูง แต่ข้อมูลของโปรเจกต์นี้มีโครงสร้างที่ชัดเจนและมีความสัมพันธ์กัน การจัดการความสัมพันธ์ของข้อมูลแชทและ User PostgreSQL เหมาะสมกว่า

### Why PostgreSQL + Prisma

PostgreSQL เป็นฐานข้อมูลที่มีประสิทธิภาพสูงและรองรับการจัดเก็บข้อมูลได้หลากหลายรูปแบบ เช่น JSON ที่เอาใช้ในส่วนการเก็บข้อความในแชท ส่วน Prisma ORM ช่วยให้จัดการฐานข้อมูล ทำให้การเขียน Schema และการจัดการ Migration ทำได้รวดเร็ว เหมาะในช่วงที่มีการเปลี่ยนโครงสร้างข้อมูลบ่อยในช่วงแรก

### Trade-offs

PostgreSQL มีขั้นตอนการติดตั้งและดูแลระบบที่ซับซ้อนกว่า MongoDB และผู้พัฒนาต้องเรียนรู้การใช้งาน Prisma เพิ่มเติม แต่ PostgreSQL มีจุดเด่นด้านความน่าเชื่อถือ ความปลอดภัย และประสิทธิภาพในการจัดการข้อมูล ทำให้เหมาะกับระบบที่ต้องรองรับข้อมูลจำนวนมากและสามารถขยายต่อได้ในอนาคต

## Decision 3: การเลือกใช้ Google Gemini API

### Context

ส่วนสำคัญของโปรเจกต์คือระบบตอบคำถาม พูดคุยกับ ai ที่สามารถเข้าใจและรองรับภาษาไทยได้ดี จึงต้องเลือกบริการ AI ที่มีประสิทธิภาพ ใช้งานง่าย และไม่มีค่าใช้จ่ายเพื่อสำหรับการศึกษาและพัฒนา

### Alternatives Considered

ตัวเลือกอื่นที่พิจารณา ได้แก่ OpenAI API ซึ่งเป็นบริการที่ได้รับความนิยมและมีประสิทธิภาพสูง แต่มีค่าใช้จ่ายเริ่มต้นในการใช้งาน และต้องเสียค่าใช้จ่ายเพิ่ม นอกจากนี้ยังมี Groq Cloud API ที่สามารถใช้งานฟรีในระดับหนึ่ง จุดเด่นคือ ตอบกลับได้เร็วมาก  แต่ผู้พัฒนาเลือกใช้ Gemini API เนื่องจากมี Free Tier ที่เพียงพอสำหรับการพัฒนาและทดสอบระบบ

### Why Google Gemini API

Google Gemini API มี Free Tier ที่เหมาะสำหรับการทดลองและพัฒนา รองรับภาษาไทยได้ดี และการพูดคุยตอบกลับก็ทำได้ดี

### Trade-offs

เวอร์ชันฟรีมีข้อจำกัดด้าน Rate Limit และจำนวน Token แต่คุณภาพของผลลัพธ์ การตอบคำถามก็สามารถทำได้ดี ทำให้ Gemini เป็นตัวเลือกที่เหมาะสมกับโปรเจกต์นี้มากที่สุด
